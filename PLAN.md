# PLAN.md — План развития калькулятора аутсорсера

---

## Обратная связь по текущей версии

### Что работает хорошо

**Математика**
- Формула расчёта корректна и методологически верна: цена считается «обратным методом», когда overhead и прибыль берутся от итоговой суммы — именно так и работает ценообразование в аутсорсинге. Это редкость в подобных инструментах.
- Мгновенный пересчёт при любом изменении параметра — никаких кнопок «Рассчитать».
- Все промежуточные переменные аккуратно изолированы в `calculations.ts` — логика не перемешана с UI.

**Дизайн и структура**
- Три карточки с итогами (цена, смена, прибыль) наверху — правильная визуальная иерархия. Самое важное видно сразу.
- Список с прогресс-барами + пончик-диаграмма — две формы подачи одного и того же, это помогает разным типам пользователей.
- История расчётов с загрузкой одним кликом — хорошая идея для сравнения вариантов.
- Адаптивность на мобильных устройствах реализована.

---

### Критичные проблемы (мешают работе)

**1. `prompt()` для сохранения расчёта**
Браузерный диалог `prompt()` — это системный попап, который:
- Выглядит чужеродно и разрушает опыт профессионального приложения
- Блокируется по умолчанию в Chrome на iOS (мобильный Safari часто игнорирует)
- Нельзя стилизовать
- Нужен: кастомный модальный диалог прямо в интерфейсе

**2. Нет защиты от ошибочных комбинаций параметров**
Если сумма `Накладные расходы + Прибыль ≥ 100%` (например, 60% + 60%), знаменатель формулы становится отрицательным и приложение молча показывает `0 ₽/час`. Пользователь не понимает, что происходит. Нужно: предупреждение красным с объяснением.

**3. GitHub Pages показывает пустую страницу**
Инструмент недоступен по публичной ссылке — основной способ поделиться им с клиентом или коллегой не работает. (Причина: токен без `workflow` scope, workflow-файл не запушен.)

---

### Важные недочёты (снижают полезность)

**4. Нет месячного расчёта**
Менеджер в полях чаще думает «сколько я заработаю с этого объекта в месяц», а не «сколько за смену». Стандартные схемы: 21 рабочий день × смена, или 176 часов. Этого показателя нет совсем.

**5. Нет расчёта на несколько человек**
Нельзя указать количество рабочих на объекте. Типичный сценарий: «У меня склад, там 8 человек, какова общая стоимость для клиента и моя прибыль?» Приходится умножать вручную.

**6. Лимит ЗП рабочего 2 000 ₽/час**
Для ИТ-аутсорсинга, квалифицированных специалистов или Москвы этот потолок занижен. Реальные ставки могут быть 3 000–8 000 ₽/час. Слайдер упирается в потолок и становится бесполезным.

**7. Нет кнопки «Сбросить значения»**
Если пользователь поэкспериментировал со значениями, вернуться к стартовым параметрам нельзя без перезагрузки страницы.

**8. В сохранённых расчётах нет даты**
Карточка показывает только название и цену. Если сохранено 5–10 расчётов, непонятно, какой из них актуальный. Дата и время помогли бы.

**9. Себестоимость не выделена как отдельный показатель**
Сейчас видна «цена для заказчика» и разбивка. Но многие финансисты хотят видеть ещё и **себестоимость** (прямые затраты до накладных и прибыли) как отдельную строку — это база для переговоров о скидках.

---

### Небольшие улучшения (качество жизни)

**10. Числа в карточках резко «прыгают»** при изменении параметров — нет анимации перехода. Добавить CSS-transition на цифры создало бы ощущение живого инструмента.

**11. Нет favicon и мета-тегов**
При открытии в браузере — стандартный значок вкладки. При репосте ссылки в Telegram/WhatsApp — нет превью-картинки.

**12. Нет возможности переименовать сохранённый расчёт**
После сохранения название нельзя изменить — только удалить и сохранить заново.

**13. Подсказка (i) видна только при ховере на десктопе**
На мобильном устройстве tooltip недоступен — нет ховера. Пользователь не узнает, что «Налоги и офис» включает офис, бухгалтерию и налоги компании.

## План развития по фазам

---

### Фаза 1 — Срочные исправления (1–2 дня)

Цель: сделать инструмент рабочим и доверенным.

| # | Задача | Что делать |
|---|---|---|
| 1.1 | **Починить GitHub Pages** | Запушить `.github/workflows/deploy.yml` с токеном, у которого есть `workflow` scope |
| 1.2 | **Убрать `prompt()`** | Заменить на встроенный модальный диалог: поле ввода + кнопки «Сохранить» / «Отмена» прямо в интерфейсе |
| 1.3 | **Валидация параметров** | Если `overhead + profit ≥ 100%`, показывать красный баннер «Сумма накладных и прибыли превышает 100% — расчёт невозможен» |
| 1.4 | **Кнопка «Сбросить»** | Маленькая ссылка-кнопка рядом с заголовком «Входные данные» — возвращает все поля к значениям по умолчанию |
| 1.5 | **Favicon и title** | Добавить favicon (эмодзи 💼 или иконка-калькулятор), прописать `<meta og:title>` и `<meta og:description>` |

## Технический долг

| Проблема | Решение |
|---|---|
| `prompt()` нарушает UX | Заменить на React-компонент `<SaveDialog>` |
| Стили разбиты между `index.css` и `<style>` в JSX | Объединить в `index.css` |
| Нет мета-тегов (SEO, sharing) | Добавить в `index.html` og:* теги |
| localStorage без версионирования | При изменении схемы данных старые записи могут сломаться — добавить `version: 1` в хранимый объект |
| Нет обработки ошибок localStorage | Если браузер в режиме инкогнито или Storage заполнен — приложение упадёт молча |

## Метрики успеха

Как понять, что инструмент становится лучше:

- **Время до первого результата** — сколько секунд нужно новому пользователю, чтобы получить цену (цель: < 30 сек)
- **Конверсия в сохранение** — какой % сессий заканчивается сохранением расчёта
- **Возвращаемость** — сколько пользователей открывают калькулятор повторно (показатель реальной ценности)
- **Переходы в Telegram** — сколько кликов по ссылкам авторов (косвенный показатель доверия)
